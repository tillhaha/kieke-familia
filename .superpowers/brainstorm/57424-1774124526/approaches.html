Shopping List — Approach Options

Three ways to architect the feature. Pick the one that fits best.

A

Lightweight — localStorage memory

Items and categories live in the DB (family-scoped). The learned item→category mapping is stored in localStorage — fast, zero DB overhead, but memory is per-browser and not shared across family members.

Pros

  • Simpler schema
  • Faster to build

Cons

  • Memory lost on new device
  • Not shared between family members
B ★ Recommended

Fully DB-backed with family sharing

All state in PostgreSQL: items, configurable categories, the learned item→category memory, and the blacklist. Optimistic UI updates for instant feedback. Memory shared across every family member.

Pros

  • Real multi-user: wife/husband share the same learned memory
  • Categories configurable per family
  • Blacklist managed in one place

Cons

  • More DB tables (4 new models)
C

B + ingredient normalization on Generate

Everything from B, plus the "Generate" logic strips quantities and units from raw ingredient strings (e.g. "2 tbsp olive oil" → "olive oil") before deduplication and blacklist matching.

Pros

  • Cleaner ingredient names on the list
  • Better deduplication across meals

Cons

  • Normalization logic is heuristic — may mangle unusual ingredient strings
  • Extra complexity
